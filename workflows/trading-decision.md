# 交易决策流程 (Trading Decision Process)

> **本文件在 Step 7 综合研判完成后、需要输出交易方案时读取。**
> 弱信号或观望结论不需要读取本文件。

---

## 8.1 前置检查（硬性门槛）

| 条件 | 行动 |
| ------ | ------ |
| 价格刻度不可读 | **禁止输出具体入场价/止损价/目标价**，改为描述条件（如"突破前高时入场""形态最低点下方止损"） |
| 信号强度为"弱" | 不输出交易方案，只给观察建议 |
| 市场状态为"混乱" | 不输出交易方案，建议观望 |
| 多周期一致性为"冲突" | 不输出交易方案，等待低周期重新与高周期同向 |

---

## 8.2 Playbook 匹配

默认先按下表快速匹配，不必先打开 `references/playbooks/INDEX.md`：

| Setup | 最少条件 | 行动 |
| ------ | --------- | ------ |
| `trend-pullback` | 趋势清晰（至少 2 次同向 BOS）+ 无反向 CHoCH + 正在回调到关键位 + 出现反转信号 | 可准备顺势方案 |
| `breakout-retest` | 关键位被动量实体突破 + 收盘站上/跌破 + 回踩突破位附近 + 出现拒绝 PA | 可准备突破回踩方案 |
| `range-reversal` | 成熟震荡区间 + 价格靠近边缘 20% + 出现反转/拒绝信号 | 可准备区间反转方案 |
| `false-breakout-reversal` | 关键位被突破 + 1~3 根 K 线内实体收回内侧 + 出现反向触发（Pin Bar/吞没/低级别反转） | 可准备假突破反向方案 |
| `flag-wedge-breakout` | 明确前置 Impulse + 形成旗形/楔形（>=5 根 K 线）+ 向趋势方向动量突破 + 收盘越过边界 | 可准备中继突破方案 |
| `无匹配` | 条件不满足或仍在中间位置 | 不给方案，建议观望 |

> 只有当需要细分"激进/保守入场"或处理 setup 边界情况时，才读取对应 Playbook 文件。

## 8.3 入场前检查（必做）

在输出交易方案前，默认先按以下简版检查；只有用户要求完整 checklist 明细时，才读取 `references/checklists/pre-trade-checklist.md`。

| 项目 | 规则 |
| ------ | ------ |
| 1. HTF 方向 | 与交易方向一致才优先；冲突则不做，震荡则降仓 |
| 2. 位置 | 在支撑附近做多、阻力附近做空；中间位置不做 |
| 3. Setup | 必须匹配 1 个 setup，且核心条件满足 |
| 4. 触发 | 必须有明确触发，不接受"感觉差不多" |
| 5. R:R | 到 T1 至少 `1.5:1` |
| 6. 事件与宏观 | 日内未来 4h 内有重大事件则不做；如有宏观数据（大盘方向/VIX/资金费率），与交易方向冲突则降仓或不做 |
| 7. 否决理由 | 至少写出 1 条潜在失败理由并评估权重 |

## 8.4 仓位与风控（内置简版）

默认先用下列规则，不必先打开 `references/risk/position-sizing.md`：

- 仓位公式：`仓位 = 风险金额 / 止损距离`
- 正常单笔风险：总资金的 `1%~2%`
- 降仓交易：总资金的 `0.5%~1%`
- 绝对上限：总资金的 `3%`
- 若有多个降级项，每个降级项让仓位再乘以 `50%`
- 若用户未提供总资金，只给风险百分比和公式，不虚构具体手数

> 只有当用户明确提供资金规模、杠杆、合约面值并要求精确仓位时，才读取 `references/risk/position-sizing.md`。

## 8.5 方案格式

```text
交易方案
├── Setup: [匹配的 Playbook 名称]
├── 方向: 做多 / 做空
├── 入场条件: [具体描述何时入场]
├── 入场价: [价格或条件]
├── 止损位: [价格] （理由：xxx）
├── 目标位1: [价格] （风险回报比 x:1）
├── 目标位2: [价格] （可选，更远目标）
├── 仓位建议: [按风险百分比与止损距离公式计算]
├── 失效条件: [什么情况下方案失效]
└── 持仓管理: [按 Playbook 差异化：趋势类(trend-pullback/flag-wedge) T1 减 30%；区间/反转类 T1 减 50%；均移止损至成本价]
```

## 8.8 复盘与回测闭环

### 单笔复盘（用户请求复盘时）

当用户提供之前的交易方案并要求复盘时，按以下格式输出：

```text
复盘: [品种] [方向] [日期]

原方案摘要:
- 入场: $xxx | 止损: $xxx | T1: $xxx | T2: $xxx

实际结果:
- 触发入场: 是/否 | 入场价: $xxx | 最终结果: T1达成/T2达成/止损/未触发

对比分析:
| 项目 | 方案预期 | 实际表现 | 偏差原因 |
|------|---------|---------|---------|
| 结构判断 | xxx | xxx | xxx |
| 关键位 | xxx | xxx | xxx |
| 触发信号 | xxx | xxx | xxx |
| 持仓管理 | xxx | xxx | xxx |

规则执行评分:
| 规则 | 是否遵守 | 说明 |
|------|---------|------|
| 按 Playbook 入场 | 是/否 | |
| Checklist 全部通过后才入场 | 是/否 | |
| 止损已设 | 是/否 | |
| 仓位按公式计算 | 是/否 | |
| T1 按 Playbook 减仓 | 是/否 | |
| 未提前/延后干预 | 是/否 | |

改进建议:
- [具体可执行的改进点，引用 Playbook/Checklist 规则]
```

> 复盘的核心不是"对不对"，而是"规则执行了没有"。即使盈利但违反规则 = 坏交易；亏损但严格执行 = 好交易。

当用户提供多笔交易记录时，输出汇总统计：

```text

胜率统计:
| Playbook | 总数 | 胜 | 负 | 未触发 | 胜率 | 平均 R |
|----------|------|----|----|--------|------|--------|
| trend-pullback | x | x | x | x | xx% | x.xR |
| breakout-retest | x | x | x | x | xx% | x.xR |
| ... | | | | | | |

常见失败原因 Top 3:
1. [原因] - 出现 N 次
2. [原因] - 出现 N 次
3. [原因] - 出现 N 次

规则改进建议:
- [基于统计数据的具体建议，如"某 Playbook 在某条件下胜率偏低，建议增加过滤条件"]
```
